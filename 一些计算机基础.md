#### 同步和异步

> 存在就有其道理 异步虽然好 但是有些问题是要用同步用来解决，比如有些东西我们需要的是拿到返回的数据在进行操作的。这些是异步所无法解决的。

也就是说，如果一段代码中有两部分是没有数据依赖关系时，可以使用异步，而有数据依赖关系的部分只能通过同步来实现！获取了前驱数据后，后续的操作才能执行。

一些零碎的思考：

1. es6中用async—await这种**异步**机制实现了并发，因为

   > 一旦遇到`await`就会先返回，等到异步操作完成，再接着执行函数体内后面的语句。

   也就是说在`async`修饰的函数内部是同步执行的，而且这是必须同步的，因为在`await`修饰的行的下面的行是依赖`await`修饰的方法等返回的数据的。

   如果调用的一个函数没有被`await`修饰，如

   ```javascript
   mainFunc(){
       let var1 = func2(...params)
       let var2=var1
       ...
   }
   
   async func2(){
       ...
       return await func3()
   }
   ```

   这种情况下`mainFunc`是不会有阻塞发生的，因为`var1`在调用`func2`时会立即得到`promise`类型返回值，然后执行下面的`var2`的复制语句。

   我突然发现其实返回`promise`这种机制是一种很巧妙的设计，是基于一种常识！

   1. 一个赋值语句是要得到值了过后才会执行下面的语句；
   2. 这个赋值语句要调用一个方法的话这个方法要执行到return后才算执行完毕。

   基于以上常识，这种设计是不需要对基本架构有什么改变的，因为它确实有返回数据，只是暂时不是真正想要的而已。
   
   > Promise模式是一种异步编程模式 。它使得我们可以先开始一个任务的执行，并得到一个用于获取该任务执行结果的**凭据对象**，而不必等待该任务执行完毕就可以继续执行其他操作
   
   这里的*凭据对象*就是promise
